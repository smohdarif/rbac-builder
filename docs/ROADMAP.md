### Phase 16: Terraform Export

**Goal:** Generate `.tf` files matching `ps-terraform-private` module patterns as an alternative to JSON.

**Why:** SAs who use Terraform for IaC can receive a Terraform deliverable instead of raw JSON + deploy.py.

**What it delivers:**
```
{customer}_terraform/
├── main.tf           ← module "project_roles" + module "default_teams"
├── variables.tf      ← standard variable definitions
├── providers.tf      ← launchdarkly provider config
└── README.md         ← how to run terraform apply
```

**New button in Deploy tab:**
```
[ 📥 Download LD Payloads ] [ 📄 Deployment Guide ] [ 📦 Delivery ZIP ] [ 🏗️ Terraform Export ]
```

**Key design decisions:**
- Uses `ps-terraform-private` module sources (not inline resources)
- Generates `key_format` / `name_format` from customer config
- Teams reference module outputs (`module.project_roles.create_flags`)
- `$${}` escaping for role attribute placeholders in HCL strings

**Files:** `services/terraform_generator.py`, `ui/deploy_tab.py`

### Phase 17: Global / Account-Level Roles

**Goal:** Add `view_teams` and `manage_personal_access_tokens` to every team — matching the sa-demo `additional_roles` pattern.

**Why:** Every sa-demo config includes these:
```hcl
additional_roles = [
  module.global_roles.manage_personal_access_tokens,
  module.global_roles.view_teams
]
```
Currently rbac-builder silently omits them.

**What changes:**
- Setup tab: "Additional Roles" section with checkboxes for account-level roles
- Payload builder: include selected keys in every team's `customRoleKeys`
- These are pre-existing LD roles — not generated by rbac-builder

**Files:** `ui/setup_tab.py`, `services/payload_builder.py`

---

### Phase 18: LD Views Support

**Goal:** Add RBAC permissions for LaunchDarkly's Views feature (saved flag filter views).

**Why:** Customers are asking for Views RBAC (see College Board Slack thread pattern). New LD feature needing coverage.

**Actions to cover (pending Gonfalon verification):**
- `createFlagView` — create a saved view
- `updateFlagView` — modify a view
- `deleteFlagView` — delete a view
- `viewFlagView` — read-only access

**Resource scope:** To be verified against `gonfalon/internal/roles/action.go` before implementation.

**UI placement:** New "👁️ Views" tab in Per-Project Permissions matrix.

**Files:** `core/ld_actions.py`, `ui/matrix_tab.py` (`PROJECT_PERMISSION_GROUPS`)

> ⚠️ **Verify actions against Gonfalon before implementing** — Views is a newer feature and action names may differ from the above.

### Phase 19: Manage Context Kinds

**Goal:** Add `Manage Context Kinds` as a default column in the project matrix.

**Why:** It's `Default=1` in the S2 template (`manage_context_kinds`) but is currently missing from the rbac-builder matrix. Every ps-terraform setup includes it.

**Actions:** `createContextKind`, `updateContextKind`, `updateAvailabilityForExperiments`
**Resource:** `proj/${roleAttribute/projects}:context-kind/*`

> **Note:** Context kind actions are already included by default in `create-flags` and `update-flags` roles (added in Phase 12 via `CONTEXT_KIND_ACTIONS_FOR_PERMISSION`). Phase 19 adds a **standalone** `Manage Context Kinds` role for teams that need it independently.

**UI placement:** Add to `PROJECT_PERMISSION_GROUPS["📊 Metrics & Pipelines"]` tab or its own group.

**Files:** `core/ld_actions.py`, `ui/matrix_tab.py`

---

### Phase 20: Deny Lists

**Goal:** Allow SAs to explicitly exclude environments, flags, or projects from a role.

**Why:** `ps-terraform-private` supports `deny_environments`, `deny_flags`, `deny_projects` variables. Some customers need to exclude specific resources (e.g., `deny_flags=["secret-*"]`).

**What changes:**
- Optional "Deny Lists" section in Setup tab
- `deny_environments`, `deny_flags`, `deny_projects` inputs (comma-separated keys)
- Payload builder: generates additional `deny` policy statements

**Example output:**
```json
{
  "effect": "deny",
  "actions": ["*"],
  "resources": ["proj/voya-web:env/*:flag/secret-*"]
}
```

**Files:** `ui/setup_tab.py`, `services/payload_builder.py`, `models/config.py`

### Phase 21: Visible Teams

**Goal:** Add `visibleTeams` role attribute to all generated teams.

**Why:** The sa-demo includes this in every team:
```hcl
role_attributes = {
  "visibleTeams" = ["default-*"]
}
```
It controls which teams a member can see. Currently silently omitted.

**What changes:**
- Setup tab: "Team Visibility Pattern" input (e.g. `{project}-*`)
- Payload builder: adds `visibleTeams` to every team's `roleAttributes`

**Files:** `ui/setup_tab.py`, `services/payload_builder.py`

---

### Phase 22: Project Admin Roles

**Goal:** Add a separate "Project Administration" section for destructive/admin actions.

**Why:** Customers need admin roles for environment management, SDK key rotation, etc. These are separate from the standard Developer/QA team roles — destructive, given to fewer people.

**Actions (from `ps-terraform-private/roles/project-administration/`):**
- `deleteProject` — delete a project (⚠️ irreversible)
- `deleteEnvironment` — delete an environment
- `updateApiKey` / `updateMobileKey` — rotate SDK keys
- `createEnvironment` — create new environments
- `updateApprovalSettings` — configure approval workflows

**UI placement:** Separate "🛡️ Project Admin" section in the matrix tab, visually distinct from standard team permissions.

**Files:** `core/ld_actions.py`, `ui/matrix_tab.py`

---

### Phase 23: Environment Tag Specifiers

**Goal:** Allow `env/*;tag1,tag2` syntax for environment filtering by tag.

**Why:** `ps-terraform-private/examples/tags.tf` uses tag-based environment grouping. Some enterprise customers organise environments by tag rather than individual keys.

**Example:**
```json
"resources": ["proj/${roleAttribute/projects}:env/*;team-platform:flag/*"]
```

**Files:** `ui/setup_tab.py`, `services/payload_builder.py`

---

### Phase 24: Flag / Segment Resource Specifiers

**Goal:** Scope roles to subsets of flags or segments using wildcard patterns.

**Why:** Enterprise customers want `flags=["feature-*"]` (only feature flags) or `deny_flags=["internal-*"]` patterns from ps-terraform.

**Example:**
```json
"resources": ["proj/${roleAttribute/projects}:env/${roleAttribute/envs}:flag/feature-*"]
```

**Files:** `ui/setup_tab.py`, `services/payload_builder.py`

---

### Phase 25: Team Management Global Roles

**Goal:** Add account-level team/member management roles.

**Why:** Enterprise customers need roles for HR/admin teams that manage LD membership, SSO, and integrations.

**Actions:**
- `manage_members` — invite, remove, modify members
- `manage_teams` — create, delete, modify teams
- `manage_integrations` — configure Slack, Datadog, etc.
- `manage_service_tokens` — manage API service tokens

**Files:** `core/ld_actions.py`, `ui/setup_tab.py` (separate "Global Roles" section)
