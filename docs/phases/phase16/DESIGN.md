## High-Level Design (HLD)

### What Are We Building and Why?

Some SA customers manage their LaunchDarkly configuration via Terraform (IaC). They want a **Terraform deliverable** they can version-control and apply with `terraform apply` — not a JSON file + Python script.

The rbac-builder currently outputs:
- JSON payload (roles + teams)
- deploy.py script (calls LD API)

Phase 16 adds a fourth delivery option:
- **Terraform package** (`.tf` files that call `terraform apply`)

### Two Approaches — Key Design Decision

The sa-demo uses **module-based Terraform** referencing ps-terraform-private local sources:

```hcl
module "project_roles" {
  source = "./roles/flag-lifecycle/per-project"   ← requires bundled modules
  ...
}
```

We choose **Option A: Inline Resources** as the primary approach.

| | Option A: Inline Resources ✅ | Option B: Module-based |
|-|-------------------------------|------------------------|
| Standalone? | ✅ — `terraform apply` immediately | ❌ — requires bundled `roles/` directory (~50 files) |
| Complexity | Low — one `main.tf` | High — must copy ps-terraform modules |
| Matches sa-demo? | ❌ — different structure | ✅ — exact match |
| Maintenance | SA owns the `.tf` directly | SA uses module abstraction |
| Best for | One-time delivery | Ongoing IaC management |

**Why Option A:** A client receiving their first RBAC setup wants to run it once and confirm it works. Module abstraction adds complexity without benefit at this stage. Option B can be a future enhancement (Phase 16b).

### Architecture

```
RBAC Builder UI (Deploy Tab)
        │
        │  Click "🏗️ Download Terraform"
        ▼
services/terraform_generator.py
  TerraformGenerator.generate_package()
        │
        ├── _build_main_tf()          → main.tf (roles + teams)
        ├── _build_providers_tf()     → providers.tf
        ├── _build_variables_tf()     → variables.tf
        └── _build_readme()           → README.md
        │
        ▼
  In-memory ZIP (bytes)
        │
        ▼
  st.download_button(data=zip_bytes)
```

### Output Package Structure

```
{customer}_terraform/
├── main.tf           ← custom roles + teams as inline resources
├── providers.tf      ← launchdarkly provider + terraform block
├── variables.tf      ← api_key variable
└── README.md         ← how to run terraform init + apply
```

### Client Workflow

```bash
unzip voya_terraform.zip
cd voya_terraform

terraform init
terraform plan   # preview changes
terraform apply  # create roles and teams in LD
```

## Detailed Low-Level Design (DLD)

### 1. `providers.tf` — always static

```hcl
terraform {
  required_providers {
    launchdarkly = {
      source  = "launchdarkly/launchdarkly"
      version = "~> 2.25"
    }
  }
  required_version = ">= 1.0"
}

provider "launchdarkly" {
  access_token = var.launchdarkly_access_token
}
```

### 2. `variables.tf` — always static

```hcl
variable "launchdarkly_access_token" {
  type        = string
  description = "LaunchDarkly API access token"
  sensitive   = true
}
```

### 3. `main.tf` — dynamically generated from payload

Three sections:

#### Section 1: Header comment
```hcl
/**
 * Generated by RBAC Builder
 * Customer  : Voya
 * Project   : voya-web
 * Generated : 2026-03-22
 * Roles     : 10
 * Teams     : 1
 */
```

#### Section 2: One `launchdarkly_custom_role` resource per role

```hcl
resource "launchdarkly_custom_role" "create_flags" {
  key              = "create-flags"
  name             = "Create Flags"
  description      = "Template role for Create Flags"
  base_permissions = "no_access"

  policy_statements {
    effect    = "allow"
    actions   = ["cloneFlag", "createFlag"]
    resources = ["proj/$${roleAttribute/projects}:env/*:flag/*"]
  }

  policy_statements {
    effect    = "allow"
    actions   = ["createContextKind"]
    resources = ["proj/$${roleAttribute/projects}:context-kind/*"]
  }

  policy_statements {
    effect    = "allow"
    actions   = ["viewProject"]
    resources = ["proj/$${roleAttribute/projects}"]
  }
}
```

> **Key:** `${roleAttribute/projects}` in JSON → `$${roleAttribute/projects}` in HCL
> (HCL uses `${}` for interpolation, so `$${}` produces a literal `${}`)

#### Section 3: One `launchdarkly_team` resource per team

```hcl
resource "launchdarkly_team" "voya_web_dev" {
  key         = "voya-web-dev"
  name        = "Voya Web: Developer"
  description = "Development team"

  custom_role_keys = [
    launchdarkly_custom_role.create_flags.key,
    launchdarkly_custom_role.update_flags.key,
    launchdarkly_custom_role.update_targeting.key,
    launchdarkly_custom_role.view_project.key,
  ]

  role_attributes {
    key    = "projects"
    values = ["voya-web"]
  }

  role_attributes {
    key    = "update-targeting-environments"
    values = ["production", "staging"]
  }

  lifecycle {
    ignore_changes = [member_ids, maintainers]
  }
}
```

### 4. Resource name sanitisation

Terraform resource names must be valid identifiers (letters, digits, underscores — no hyphens).

```python
# role key:  "create-flags"          → TF resource name: "create_flags"
# team key:  "voya-web-dev"          → TF resource name: "voya_web_dev"
# rule:      replace hyphens with underscores

def tf_resource_name(key: str) -> str:
    return key.replace("-", "_")
```

### 5. `TerraformGenerator` class

```python
class TerraformGenerator:
    def __init__(self, payload: DeployPayload, project_key: str):
        self.payload     = payload
        self.project_key = project_key
        self.slug        = payload.customer_name.lower().replace(" ", "_")

    def generate_package(self) -> bytes:
        """Build ZIP and return as bytes."""

    def _build_providers_tf(self) -> str:
        """Returns static providers.tf content."""

    def _build_variables_tf(self) -> str:
        """Returns static variables.tf content."""

    def _build_main_tf(self) -> str:
        """
        Dynamically generates main.tf from payload.
        Sections: header comment → custom roles → teams
        """

    def _role_to_hcl(self, role: dict) -> str:
        """Convert one role dict to a launchdarkly_custom_role HCL block."""

    def _team_to_hcl(self, team: dict, role_keys: Set[str]) -> str:
        """Convert one team dict to a launchdarkly_team HCL block."""

    def _build_readme(self) -> str:
        """Returns Terraform usage instructions."""
```

### 6. HCL generation helpers

#### `_role_to_hcl(role)` logic

```
resource_name = tf_resource_name(role["key"])

lines = [
  f'resource "launchdarkly_custom_role" "{resource_name}" {{',
  f'  key              = "{role["key"]}"',
  f'  name             = "{role["name"]}"',
  f'  description      = "{role.get("description", "")}"',
  f'  base_permissions = "no_access"',
  "",
]

FOR each stmt in role["policy"]:
  lines += [
    "  policy_statements {",
    f'    effect    = "{stmt["effect"]}"',
    f'    actions   = {hcl_list(stmt["actions"])}',
    f'    resources = {hcl_list_escaped(stmt["resources"])}',
    "  }",
    "",
  ]

lines.append("}")
```

#### `_team_to_hcl(team, role_keys)` logic

```
resource_name = tf_resource_name(team["key"])

# Build custom_role_keys as TF references
role_refs = [
  f"launchdarkly_custom_role.{tf_resource_name(k)}.key"
  for k in team["customRoleKeys"]
  if k in role_keys   # only reference roles we generated
]

lines = [
  f'resource "launchdarkly_team" "{resource_name}" {{',
  f'  key         = "{team["key"]}"',
  f'  name        = "{team["name"]}"',
  f'  description = "{team.get("description", "")}"',
  "",
  "  custom_role_keys = [",
]

FOR ref in role_refs:
  lines.append(f"    {ref},")

lines += [
  "  ]",
  "",
]

FOR attr in team["roleAttributes"]:
  lines += [
    "  role_attributes {",
    f'    key    = "{attr["key"]}"',
    f'    values = {hcl_list(attr["values"])}',
    "  }",
    "",
  ]

lines += [
  "  lifecycle {",
  "    ignore_changes = [member_ids, maintainers]",
  "  }",
  "}",
]
```

### 7. HCL formatting helpers

```python
def hcl_list(values: List[str]) -> str:
    """Format a Python list as an HCL list."""
    items = ", ".join(f'"{v}"' for v in values)
    return f"[{items}]"

def hcl_list_escaped(resources: List[str]) -> str:
    """
    Format resources for HCL, escaping ${} placeholders.
    JSON: "proj/${roleAttribute/projects}:env/*:flag/*"
    HCL:  ["proj/$${roleAttribute/projects}:env/*:flag/*"]
    """
    escaped = [r.replace("${", "$${") for r in resources]
    return hcl_list(escaped)
```

## Pseudo Logic

### `_build_main_tf()`

```
FUNCTION _build_main_tf():

  sections = []

  # Header
  sections.append(header_comment(customer, project_key, date, role_count, team_count))

  # Custom roles
  sections.append("# ── Custom Roles ──────────────────────────────────────────────────────")
  FOR each role in payload.roles:
    sections.append(_role_to_hcl(role))

  # Teams
  sections.append("# ── Teams ─────────────────────────────────────────────────────────────")
  role_keys = {role["key"] for role in payload.roles}
  FOR each team in payload.teams:
    sections.append(_team_to_hcl(team, role_keys))

  RETURN "\n\n".join(sections)
```

### `hcl_list_escaped(resources)`

```
FUNCTION hcl_list_escaped(resources):

  escaped = []
  FOR each resource in resources:
    # JSON uses ${roleAttribute/...}
    # HCL uses $${roleAttribute/...} to produce a literal ${}
    escaped.append(resource.replace("${", "$${"))

  RETURN "[" + ", ".join(f'"{r}"' for r in escaped) + "]"
```

## Test Cases

**Test file:** `tests/test_terraform_generator.py`

### Group 1: Package Output

#### TC-TF-01: Returns valid ZIP bytes
```
GIVEN: a DeployPayload with roles and teams
WHEN:  generate_package() called
THEN:  result is bytes, opens as valid ZipFile
```

#### TC-TF-02: ZIP contains required files
```
GIVEN: customer_name="Voya"
WHEN:  generate_package() called
THEN:  ZIP contains:
  "voya_terraform/main.tf"
  "voya_terraform/providers.tf"
  "voya_terraform/variables.tf"
  "voya_terraform/README.md"
```

### Group 2: providers.tf and variables.tf

#### TC-TF-03: providers.tf contains launchdarkly provider
```
THEN: content contains "launchdarkly/launchdarkly"
      content contains "required_version"
      content contains "var.launchdarkly_access_token"
```

#### TC-TF-04: variables.tf declares api token variable
```
THEN: content contains "launchdarkly_access_token"
      content contains "sensitive   = true"
```

### Group 3: main.tf — Roles

#### TC-TF-05: Each role generates a resource block
```
GIVEN: payload with role key="create-flags"
THEN:  main.tf contains:
  'resource "launchdarkly_custom_role" "create_flags"'
  'key              = "create-flags"'
  'base_permissions = "no_access"'
```

#### TC-TF-06: Hyphens in resource names converted to underscores
```
GIVEN: role key="update-targeting"
THEN:  resource block starts with:
  'resource "launchdarkly_custom_role" "update_targeting"'
  (NOT "update-targeting" — invalid HCL)
```

#### TC-TF-07: ${roleAttribute/...} placeholders escaped in HCL
```
GIVEN: role with resource "proj/${roleAttribute/projects}:env/*:flag/*"
THEN:  main.tf contains:
  "proj/$${roleAttribute/projects}:env/*:flag/*"
  ($ doubled to produce literal ${ in HCL)
  NOT "proj/${roleAttribute/projects}..."  ← would be HCL interpolation error
```

#### TC-TF-08: Policy statements match JSON source
```
GIVEN: role with 3 policy statements
THEN:  main.tf contains 3 "policy_statements {" blocks for that role
       actions list matches JSON source exactly
```

### Group 4: main.tf — Teams

#### TC-TF-09: Each team generates a resource block
```
GIVEN: team key="voya-web-dev"
THEN:  main.tf contains:
  'resource "launchdarkly_team" "voya_web_dev"'
  'key         = "voya-web-dev"'
```

#### TC-TF-10: Team references roles by TF resource reference
```
GIVEN: team with customRoleKeys=["create-flags", "update-flags"]
THEN:  main.tf contains in custom_role_keys:
  "launchdarkly_custom_role.create_flags.key"
  "launchdarkly_custom_role.update_flags.key"
  (NOT string literals "create-flags" — uses TF references for dependency graph)
```

#### TC-TF-11: Team role_attributes blocks generated correctly
```
GIVEN: team with roleAttributes:
  [{"key": "projects", "values": ["voya-web"]},
   {"key": "update-targeting-environments", "values": ["production"]}]
THEN:  main.tf contains:
  role_attributes { key = "projects" values = ["voya-web"] }
  role_attributes { key = "update-targeting-environments" values = ["production"] }
```

#### TC-TF-12: lifecycle ignore_changes block present on all teams
```
THEN:  every team resource contains:
  "lifecycle {"
  "ignore_changes = [member_ids, maintainers]"
```

### Group 5: Edge Cases

#### TC-TF-13: Team references only roles present in payload
```
GIVEN: team with customRoleKeys that includes a key NOT in payload.roles
THEN:  that key is omitted from custom_role_keys in the TF output
       (prevents broken TF references)
```

#### TC-TF-14: main.tf is valid HCL syntax (no ${} interpolation errors)
```
GIVEN: any payload with roleAttribute placeholders
WHEN:  main.tf is inspected for unescaped ${
THEN:  no unescaped "${" appears inside string literals
       (all role attribute placeholders use "$${")
```

#### TC-TF-15: main.tf contains no duplicate resource blocks
```
GIVEN: payload with 10 roles and 2 teams
THEN:  main.tf has exactly 10 custom_role resource blocks
       and exactly 2 team resource blocks
```
