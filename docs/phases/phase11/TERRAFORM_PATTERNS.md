## 2. Environment Handling Patterns

### ✅ Environment Scoping via Role Attribute Pattern
```hcl
# Each env role uses a per-permission attribute placeholder
environments = ["*;${roleAttribute/update-targeting-environments}"]

# Teams fill in exact environment keys via roleAttributes
role_attributes {
  key    = "update-targeting-environments"
  values = ["production", "staging"]
}
```

**Status:** Implemented — using the **role attribute pattern** (sa-demo approach).

> ⚠️ **Decision note:** We evaluated `{critical:true}/{critical:false}` wildcard filtering
> (Phase 12 design) but chose the explicit role attribute pattern instead. See
> [Role Attribute Pattern docs](../../role-attribute-pattern/HLD.md) for rationale.

**Why role attribute pattern over `{critical:true}` filter:**
- No LD-side environment property setup required
- Explicit: you can read exactly which envs each team has access to
- Per-role scoping: different env lists for update-targeting vs apply-changes
- Proven pattern from ps-terraform-private sa-demo

**Implemented in:**
- `core/ld_actions.py` → `PERMISSION_ATTRIBUTE_MAP` (kebab-case attr keys)
- `services/payload_builder.py` → `RoleAttributePayloadBuilder`
- `tests/test_critical_environments.py` → full test coverage

## 3. Role Configuration Patterns

### 🔲 Granular Permission Toggles

Current RBAC Builder has combined permissions. Terraform has separate toggles:

| Current (Combined) | Terraform (Granular) |
|--------------------|---------------------|
| Create Flags | `create_flags`, `update_flags` (separate) |
| Manage Segments | `create_segments`, `update_segments`, `delete_segments` |
| - | `manage_releases` vs `manage_release_pipelines` |
| - | `update_flag_variations` (separate from update_flags) |
| - | `manage_context_kinds` |
